# TeleRAG-Agent: Agentic Design Deep-Dive

*Samsung EnnovateX AX Hackathon — Problem 10: RAG-based Future-Ready Telecom RAN Assistant*

---

## Why Agentic RAG?

A naive RAG system does: **retrieve → generate**. This fails for telecom because:

1. **Multi-step questions**: "Compare DRX in LTE vs 5G NR" requires two separate retrievals and a synthesis step.
2. **Unknown gaps**: A single retrieval may miss the relevant section. The system needs to detect when its answer is weak and try again.
3. **Confidence awareness**: A network engineer cannot use an answer they cannot trust. The system must be able to say "I'm not sure — here's what I found, but you should verify."

Our LangGraph implementation addresses all three.

---

## The PLAN → RETRIEVE → GENERATE → REFLECT Loop

### Why LangGraph?

LangGraph compiles a **state machine** — not an agent loop that calls `llm.invoke()` in a Python for-loop. This means:

- Each node is a pure function `(AgentState) → AgentState`
- The routing logic (which node runs next) is separated from the node logic
- The graph can be inspected, traced, and debugged deterministically
- The maximum iteration count is enforced by the conditional edge, not by fragile `if` statements inside the loop

### PLAN Node

**Input:** raw query string  
**Output:** `query_type`, `sub_queries`, `tools_to_use`

The PLAN node does two things:

1. **Query classification** — Uses regex + keyword matching (NOT the LLM) to classify into:
   - `spec_lookup` — "What is X in 3GPP?" → pure retrieval
   - `fault_diagnosis` — "Why is handover failing?" → alarm tool + retrieval
   - `kpi_analysis` — "What is RSRP threshold?" → KPI tool + retrieval
   - `comparison` — "Compare X and Y" → dual retrieval
   - `open_ended` — general technical discussion

   *Design decision: we use pattern matching, not the LLM, for classification. Why? The LLM is expensive (30-90 seconds on T4). Query classification is deterministic and fast. We should not burn LLM budget on routing.*

2. **Query decomposition** — For complex queries, splits into 2-3 sub-queries that are each run independently through retrieval and then context-merged.

### RETRIEVE Node

**Input:** query + sub_queries  
**Output:** `retrieved_context`, `sources`

Runs the full hybrid pipeline:
- **Dense**: BGE-large cosine similarity (semantic understanding)
- **Sparse**: BM25 term overlap (exact terminology like "HARQ" or "RLC AM")
- **KG**: Section heading traversal (boosts chunks from relevant spec sections)
- **RRF**: Merges all three without needing score calibration
- **Reranker**: Cross-encoder rescores the merged list to catch false positives

*Design decision: Why 3 retrieval methods?*  
- Dense alone misses exact 3GPP terminology ("TS 38.321 §5.4.2" doesn't embed well)
- Sparse alone misses paraphrases ("discontinuous reception" vs "DRX")
- KG alone is too coarse (section level, not chunk level)
- Each captures a different signal; RRF combines them without needing weights

### GENERATE Node

**Input:** query + context  
**Output:** `answer`, `citations`

Builds a structured prompt in the exact format the LoRA adapter was fine-tuned on:
```
### Question:
[Retrieved context...]
[User question]

### Answer:
```

The generation is deterministic (temperature=0, do_sample=False) to ensure reproducibility.

Citations are injected by post-processing the retrieved source list, not generated by the LLM. This prevents hallucinated citations.

### REFLECT Node

**Input:** answer + sources + iteration count  
**Output:** `confidence`, routing decision

Confidence is a **heuristic composite score**, not an LLM self-evaluation. We deliberately avoid asking the LLM "are you confident?" because:
- The LLM cannot accurately judge its own correctness
- It costs another generation call
- It adds latency

Instead, confidence is computed from observable signals:
```python
confidence = (
    0.3 * min(len(sources) / 5, 1.0)        # source count
  + 0.3 * max_rerank_score                   # quality of top source
  + 0.2 * min(len(answer) / 200, 1.0)        # answer length (not empty)
  + 0.2 * citation_bonus                     # has specific clause citation
)
```

Routing:
- ≥ 0.8: **done** — high confidence, exit
- 0.5–0.8: **retrieve** — expand query, re-retrieve (max 3 total iterations)
- < 0.5: **clarify** — tell user we're unsure

## What Worked

1. **RRF fusion** — Combining dense + sparse + KG consistently outperformed any single method. BM25 alone finds "HARQ process" but misses "hybrid ARQ". Dense alone finds related concepts but misses exact clause references.

2. **Cross-encoder reranking** — The cross-encoder dramatically improved Recall@1 (precision of top-1 result). Without it, the top-1 result was often a definitions/abbreviations section, not the actual procedure.

3. **Heuristic confidence** — Routing decisions based on observable signals are fast (~0ms) and predictable. The system reliably re-retrieves when sources are weak.

4. **Lazy model loading** — Loading the LLM on first query (not at startup) keeps the Gradio UI responsive during initialization.

5. **Gradio version-aware UI** — Using `inspect` to detect available kwargs made the UI deploy identically on Gradio 3.x (local dev), 5.x (Kaggle T4), and 6.x (latest).

---

## What Didn't Work / Trade-offs

1. **vLLM** — We attempted vLLM for faster inference but hit `torch._utils` incompatibilities with PyTorch 2.10. We fell back to transformers + 4-bit bitsandbytes quantization. Latency is ~45-90s/query on T4 instead of the target ~8s.

   *Mitigation: The UI shows a "Thinking..." placeholder and the thinking trace gives the user something to read while waiting. For a production system, we would use a dedicated inference server.*

2. **Sentence-transformers pinning** — Pinning to `==2.7.0` caused crashes on PyTorch 2.10 (removed `torch._utils` internals). Fixed by upgrading to latest (5.6.0) where `torchcodec` is optional.

3. **LoRA fine-tuning accuracy** — Initial accuracy was 44% due to a training/inference prompt mismatch. After fixing the template to match the `### Question / ### Answer` format, accuracy improved to 72% on the 50-question golden set. We did not hit the 80% target — this is primarily because:
   - Fine-tuning only on MCQ format; open-ended questions are harder
   - Golden set questions reference specs we may not have fully indexed
   - The base model (LLaMA-3-8B-Tele-it) already has strong telecom knowledge from pre-training, which reduces the LoRA delta

4. **KG is heading-level only** — Our KG has ~200 nodes (section headings) rather than concept-level entities. A production system would extract entities like `HARQ`, `RLC AM`, `gNB` and link them across specs. This is a known limitation for the current timeline.

## Design Decisions Summary

| Decision | Choice | Rationale |
|---|---|---|
| Agent framework | LangGraph | Deterministic state machine; inspectable; not a fragile for-loop |
| Query routing | Pattern matching (not LLM) | Fast, deterministic, no LLM cost for routing |
| Confidence scoring | Heuristic (not LLM self-eval) | Reliable, fast, no extra generation |
| Citation injection | Post-processing (not generated) | Prevents hallucinated citations |
| Retrieval fusion | RRF over learned weights | Works without calibration; robust to new query distributions |
| Quantization | 4-bit NF4 (bitsandbytes) | Fits 8B model in 15.6GB T4 VRAM with inference quality preservation |
| Local Qdrant | File path, not server | Zero infrastructure on Kaggle; no Docker required |
